# Keycloak Theme

This project is a separate Angular app that renders the custom Keycloak login theme for Expense Tracker.

It is built with `@keycloakify/angular` and Vite, and it shares the workspace theme tokens and Material skinning so the login screens stay aligned with the main UI.

## Local dev

This project now has two useful local-dev modes:

- Vite shell preview for quick CSS/layout checks when no `kcContext` is present
- Storybook for mocked Keycloak pages outside Keycloak, which is the workflow closest to the Keycloakify quickstart

### Vite preview shell

Run the Keycloak theme dev server from the workspace root:

```bash
npm run serve:keycloak-theme
```

Equivalent Nx command:

```bash
.\node_modules\.bin\nx.cmd run keycloak-theme:serve
```

Vite will usually serve the app at `http://localhost:5173`.

By default, local dev starts the theme app without a real `window.kcContext`, so you will see the lightweight preview shell from [src/no-context.component.ts](./src/no-context.component.ts) rather than a full Keycloak login page.

That is expected.

### Storybook outside Keycloak

Run Storybook from the workspace root:

```bash
npm run storybook:keycloak-theme
```

Equivalent Nx command:

```bash
.\node_modules\.bin\nx.cmd run keycloak-theme:storybook
```

Storybook usually serves on `http://localhost:6006` and renders real Keycloak pages using mocked `kcContext`.

The initial setup includes a mocked [login.ftl story](./src/login/pages/login/login.stories.ts), so you can iterate on the actual login screen without packaging the theme or booting Keycloak.

To add more page stories with the Keycloakify CLI:

```bash
npm run add-story:keycloak-theme
```

That command uses `keycloakify add-story --project projects/keycloak-theme` and writes the new story under `src/login/pages/...`.

### Running against real Keycloak

To see the actual Keycloak pages inside Keycloak, you still need one of these:

- run the packaged theme inside a local Keycloak instance
- inject or mock a `kcContext` during development

This repo now ships all three layers:

- a lightweight Vite preview shell
- Storybook mocked-page previews outside Keycloak
- the real packaged theme flow for deployment

## Build and package

Build the frontend bundle only:

```bash
npm run build:keycloak-theme
```

Build and package the deployable Keycloak theme:

```bash
npm run package:keycloak-theme
```

The package step runs the Vite build first and then packages the Keycloak theme artifacts through `keycloakify`.

Build the Storybook static preview:

```bash
npm run build-storybook:keycloak-theme
```

## Important files

- [project.json](./project.json): Nx targets for `serve`, `build`, and `package-theme`
- [vite.config.mts](./vite.config.mts): Vite + Keycloakify setup, theme name, and default `UI_THEME`
- [src/main.ts](./src/main.ts): chooses between preview shell and real Keycloak bootstrap
- [src/no-context.component.ts](./src/no-context.component.ts): preview shell shown when `kcContext` is missing
- [.storybook/main.ts](./.storybook/main.ts): Storybook config for outside-Keycloak page previews
- [src/login/KcPageStory.ts](./src/login/KcPageStory.ts): Storybook harness that injects mocked `kcContext`
- [src/login/template/template.component.ts](./src/login/template/template.component.ts): top-level Keycloak page frame and theme switching
- [src/login/pages/login/login.component.ts](./src/login/pages/login/login.component.ts): custom login page component
- [src/login/pages/login/login.stories.ts](./src/login/pages/login/login.stories.ts): initial mocked login-page stories
- [src/styles.css](./src/styles.css): Keycloak-specific layout plus shared theme imports

## Generated vs editable files

Safe to edit:

- `projects/keycloak-theme/src/**`, except generated files
- `projects/keycloak-theme/vite.config.mts`
- `projects/keycloak-theme/project.json`

Do not edit manually:

- [src/kc.gen.ts](./src/kc.gen.ts)

That file is generated by `keycloakify` and will be recreated.

## Theme alignment

The Keycloak theme intentionally shares visual infrastructure with the main app:

- shared theme constants from `@expense-tracker-ui/shared/theme`
- shared theme tokens from `libs/shared/theme`
- shared Material theme output from `src/styles/theme.scss`
- shared Material overrides from `src/styles/material.css`

That keeps buttons, inputs, tokens, and theme names aligned without making the Keycloak app depend on the main app runtime.
